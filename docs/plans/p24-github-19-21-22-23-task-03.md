## Change description

### 문제 상세

`get_sessions_since()`의 SQL:
```sql
SELECT ... FROM sessions WHERE start_time >= ?1 ORDER BY start_time
```

- `?1` = `"2026-04-10"` (사용자 입력)
- `start_time` = `"2026-04-10T09:00:00+09:00"` (RFC3339)

SQLite 문자열 비교:
- `"2026-04-10" < "2026-04-10T..."` → 이 세션은 포함됨 ✅
- `"2026-04-09T15:00:00Z"` (UTC 기준 전날이지만 KST로는 4/10) → `>= "2026-04-10"`은 false → **누락** ❌

반대 방향도 문제:
- `"2026-04-10T23:30:00+09:00"` → 문자열로는 `>= "2026-04-10"` true
- 하지만 이게 UTC로는 4/10 14:30이라 포함이 맞음

**핵심 문제**: KST(+09:00) 세션의 `start_time`이 UTC 기준으로 전날 날짜를 가지면 `YYYY-MM-DD` 비교에서 누락됨.

### 수정 단계

#### Step 1 — `get_sessions_since()`에서 since 파라미터 정규화

짧은 날짜 형식(`"2026-04-10"`)이 들어오면 `"2026-04-10T00:00:00+00:00"`으로 변환.
이미 ISO 형식이면 그대로 사용.

```rust
pub fn get_sessions_since(&self, since: &str) -> Result<Vec<WikiSessionMeta>> {
    // 날짜만 입력된 경우 UTC 자정으로 정규화
    let since_normalized = if since.len() == 10 && since.chars().nth(4) == Some('-') {
        format!("{}T00:00:00+00:00", since)
    } else {
        since.to_string()
    };
    let mut stmt = self.conn().prepare(
        "SELECT ... FROM sessions WHERE start_time >= ?1 ORDER BY start_time",
    )?;
    // ... since_normalized 사용
}
```

이렇게 하면 `"2026-04-10T00:00:00+00:00"` vs `"2026-04-09T15:00:00Z"` → UTC 기준 정확한 비교.

단, SQLite는 문자열 비교이므로 `+00:00`과 `Z`의 사전순 차이에 주의.
`"2026-04-09T15:00:00Z"` < `"2026-04-10T00:00:00+00:00"` → 제외 ✅ (올바름)

다양한 타임존의 `start_time`을 가진 세션을 생성하고 since 필터링이 정확한지 검증.

## Risks

- SQLite 문자열 비교는 `datetime()` 함수와 다름. RFC3339 문자열의 사전순 비교가
  시간순과 항상 일치하지는 않음 (타임존 오프셋 부호에 따라).
  예: `"+09:00"` > `"+00:00"` > `"-05:00"` 사전순으로 올바름.
  하지만 `"Z"` < `"+"` 이므로 UTC를 `Z`로 표기한 세션과 `+00:00`으로 표기한 세션의
  비교가 다를 수 있음.
- **안전한 대안**: SQL에서 `datetime(start_time)` 함수를 사용하여 UTC로 통일 비교.
  `WHERE datetime(start_time) >= datetime(?1)` — SQLite datetime()은 RFC3339을 파싱함.
  이 방법이 더 정확하지만, FTS5와 달리 인덱스 활용이 안 될 수 있음.
  sessions 테이블은 규모가 작으므로(~1000행) 성능 영향 무시 가능.
