## Change description

### 문제 상세

**관찰 결과** (세션 `6e5dd35d`, 351회 compact):
- JSONL: 11,235줄, user 490개, assistant 4,871개
- Python 시뮬레이션(파서 로직 재현): 5,361 turns 수집
- DB: user 1, assistant 34 — **전체의 0.6%만 저장됨**

**근본 원인 분석**:

1. `insert_session()`이 `INSERT OR IGNORE` → 이미 세션 존재 시 메타 업데이트 안 됨
2. `insert_turn()`도 `INSERT OR IGNORE` + `UNIQUE(session_id, turn_index)` 제약
3. 최초 인제스트: 세션의 마지막 상태(compact 이후 일부 turn)만 존재
4. `--force` 재인제스트: `delete_session_full()` 후 재삽입하지만, FTS는 미삭제(#23)
5. compact 없이도: 파서가 `turns.len() as u32`로 인덱스 부여하므로 UNIQUE 충돌은 없음

**실제 원인**: 최초 인제스트 시점에 세션이 아직 진행 중(오픈)이었을 수 있음.
이후 세션이 종료되고 compact가 반영된 JSONL로 재인제스트할 때,
세션이 이미 존재하여 `INSERT OR IGNORE`로 스킵됨.

### 수정 단계

#### Step 1 — `last-prompt` 엔트리 구조 파악

조사 결과:
```json
{"type": "last-prompt", "lastPrompt": "<사용자 마지막 입력 텍스트>", "sessionId": "..."}
```

`lastPrompt`는 이전 대화 메시지 배열이 아니라 마지막 사용자 입력 텍스트.
compact 전 대화는 JSONL에 이미 별도의 user/assistant 엔트리로 기록되어 있음.
파서는 이미 이들을 수집하고 있음 (시뮬레이션에서 490 user turns 확인).

#### Step 2 — 오픈 세션 재인제스트 로직 개선

`ingest.rs`의 오픈 세션 재인제스트 경로 확인:

```rust
// ingest.rs:264-267 (현재)
Ok(true) => {
    // 기존 레코드 삭제 후 재인제스트
    if let Err(e) = db.delete_session_full(session_id_hint) {
```

이 경로는 `is_session_open()` == true일 때만 동작.
이미 close된 세션은 `session_exists()` → true → skip.

**수정**: 세션이 존재하지만 turn 수가 현재 파싱 결과보다 적으면 재인제스트.
또는 더 단순하게: `--force` 없이도 파싱된 turn 수와 DB turn 수를 비교하여
차이가 크면 자동으로 재인제스트.

#### Step 3 — 구현

`ingest.rs`의 중복 체크 로직에 turn 수 비교 추가:

```rust
// 기존: session_exists → skip
// 변경: session_exists → turn 수 비교 → 차이 크면 재인제스트
Ok(true) if !force => {
    let db_turn_count = db.count_turns_for_session(&session.id)?;
    if session.turns.len() > db_turn_count * 2 {
        // 파싱 결과가 DB보다 2배 이상 많으면 재인제스트
        tracing::info!(
            session = &session.id,
            db_turns = db_turn_count,
            parsed_turns = session.turns.len(),
            "re-ingesting session with significantly more turns"
        );
        db.delete_session_full(&session.id)?;
        // 계속 진행하여 재인제스트
    } else {
        // 정상 스킵
        return;
    }
}
```

#### Step 4 — `count_turns_for_session()` 메서드 추가

`session_repo.rs`에 추가:
```rust
pub fn count_turns_for_session(&self, session_id: &str) -> Result<usize> {
    let count: i64 = self.conn().query_row(
        "SELECT COUNT(*) FROM turns WHERE session_id = ?1",
        [session_id],
        |r| r.get(0),
    )?;
    Ok(count as usize)
}
```

## Risks

- **turn 수 비교 임계값**: 2배 기준이 적절한지 판단 필요.
  compact가 1-2번인 세션은 turn 수 차이가 작을 수 있음.
  대안: turn 수 차이 절대값 기준 (예: 10개 이상 차이).
- **대량 세션 재인제스트 위험**: turn 수 비교를 너무 공격적으로 하면
  정상 세션도 매번 재인제스트될 수 있음. 보수적 임계값 사용.
- **ingest.rs는 P23 scope boundary에서 수정 금지였으나**, 이 task는 P24이므로 적용 가능.
  다만 ingest의 기존 orchestration 로직(분류, 벡터 임베딩 등)은 건드리지 않음.
- `count_turns_for_session()` 추가 시 session_repo.rs에 `impl Database` 블록 확장.
  Task 01과의 충돌 가능성 → Task 01 완료 후 또는 다른 메서드 범위이므로 병렬 가능.
